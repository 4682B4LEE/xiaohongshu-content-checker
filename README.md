## 为什么做这个 Skill

说出来你可能不信，我是因为**被封号了**才做这个工具的。

之前做了一款小程序，想找个地方推广。研究了一圈，发现小红书的自然流推荐机制对低粉新号极其友好，于是把小红书当成首选阵地。结果因为没注意外链引流的问题，直接收到了封号处罚。

更坑的是，有些词我们日常觉得完全没问题，但在平台机审的背景下，很容易被限流。比如我精心准备的笔记，改了好几版，最后虽然合规了，但错过了最佳发布时间，还被降权，传播效果大打折扣。

折腾了两个月，踩了无数坑，也写出了多篇上万阅读、千赞的笔记。我把自己和小红书审核申诉的实战经验，加上对平台规则的理解，做成了这个 Skill。

**目标很简单：帮你在发笔记前，把那些"必死"的雷排掉。**

---

## 这个 Skill 能帮你解决什么

你是不是也遇到过这些问题：

- 用了付费敏感词工具调整，流量还是没明显提升？
- 笔记被限流，官方只给一个模糊的风险提示，不告诉你具体是封面有问题还是内容有问题？
- 反复修改、重新审核，最后合规了，但错过了最佳发布时间，还被降权？
- 单纯检查敏感词，发现根本解决不了限流问题？

这些问题我都经历过。所以这个 Skill 不只是帮你查敏感词，而是采用 **"先定位、后检测、再优化"** 的标准工作流：

- **先定位**：分析你的账号人设和内容领域，判断笔记和定位是否匹配
- **后检测**：基于最新平台规则，识别敏感词、导流风险、价值观导向问题、AIGC 合规等
- **再优化**：给出结合你账号定位的专业修改建议，而不是千篇一律的替换词

## 使用流程

Skill 会根据用户输入自动识别意图，选择最合适的检测模式：

1. **意图识别**：分析用户输入，匹配唤醒词或 CLI 指令
2. **模式路由**：自动选择极速排雷 / 完整检测 / 批量检测 / 视频专项
3. **风险检测**：加载对应知识库，执行关键词匹配 + 语义分析（双引擎）
4. **输出报告**：先结论后证据，只列命中项，只改高风险和中风险

## 设计规范

- **评分唯一来源**：所有模式共用 `risk-rubric.md`，避免规则碎片化
- **双引擎检测**：关键词匹配 + 语义分析，互补验证
- **先结论后证据**：用户第一眼看到风险等级和核心问题
- **只列命中项**：没有风险的项目不输出，避免信息过载
- **可选增强**：标题/标签/封面/评论区建议仅在用户回复"继续优化"后生成
- **不绝对判罚**：使用"存在推荐受限风险""可能触发审核"等建议性表述
- **防御性设计**：Anti-Jailbreak 策略，拒绝恶意探测、变体绕过、反向诱导

## 设计哲学

做内容审核类的 Skill，最大的敌人不是规则复杂，而是 **AI 的发散性**。

如果你单纯依赖大模型自由发挥，一定会遇到这种尴尬：同一篇笔记检测三次，得出三个完全不同的结论。有时候 AI 会过度敏感，把正常的表达也标红；有时候又会漏掉明显的导流风险。

所以这类 Skill 的核心设计思路是 **"规则的结构化"与"约束的前置"**：

1. **把平台红线变成外挂的客观标准**
   - 敏感词库、评分规则、定位指南全部拆成独立的 reference 文件
   - AI 不是"凭感觉"判断，而是对照明确的规则逐项核对

2. **用强规则对抗 AI 的幻觉**
   - 每个风险项有明确的分值（+1 到 +4）
   - 每个分值对应明确的颜色标识（🔴🟠🟡）
   - 总分区间对应明确的等级和建议

3. **赋予 AI 极强的"边界感"**
   - 先核对敏感词 → 再看人设匹配 → 最后给出修改建议
   - 不输出没有风险的项目，不编造不存在的规则

这个思路不仅适用于小红书内容审核，也可以迁移到任何需要"严谨性"的 AI 工作流中。**用强规则去对抗 AI 的发散，才能让它真正变成一个稳定、可靠、每次输出标准一致的"金牌审核员"。**
