# 小红书内容检测 Skill 增强优化 Spec

## Why

当前的小红书内容检测 Skill 已经具备5步SOP工作流程、CES算法洞察、营销浓度评估、变体敏感词识别等核心能力。但结合 chatgpt.md 的方案，在**结构化输出**（内容定位卡、固定输出模板）、**检测维度完整性**（封面文案、评论区）、**风险分类精细化**（违规风险 vs 流量风险）、**特殊行业适配**等方面仍有提升空间。本次优化旨在借鉴 chatgpt.md 的精华，进一步完善检测维度和输出规范。

## What Changes

- **新增内容定位卡**：在检测前强制提炼账号定位、目标人群、笔记类型、发布目的、核心卖点、用户动作、匹配度
- **明确区分违规风险与流量风险**：将风险分为"违规风险"（审核、折叠、封号）和"流量风险"（定位偏离、广告化、缺少痛点、标签过泛）
- **新增封面文案检测维度**：封面文字控制在8-16字、一眼看懂、避免堆叠卖点
- **新增评论区与互动引导检测**：专门检测评论区导流、诱导私信、诱导点赞收藏、诱导交易
- **补充特殊行业处理规则**：新增AI工具/SaaS、医美/减肥/健康、教育/副业/赚钱、品牌种草/商业合作的特殊处理规则
- **标准化输出模板**：固定9步输出模板（总体结论→定位卡→问题摘要→敏感词→匹配度→自然流→修改建议→优化版本→CheckList）
- **完善信息不足处理流程**：明确信息不足时最多问3-5个问题，用户不想补充也能继续检测
- **新增修改原则**：明确5条修改原则（降低风险、增强匹配、提升自然感、保留核心卖点、给出可直接替换版本）

## Impact

- Affected specs: 小红书内容检测 Skill 的检测维度、输出格式、特殊行业规则
- Affected code: SKILL.md, README.md, references/PLATFORM_RULES.md, references/CONTENT_EXAMPLES.md

## ADDED Requirements

### Requirement: 内容定位卡

The system SHALL generate a structured content positioning card before detection:

#### Scenario: 建立内容定位卡
- **WHEN** 用户提供账号信息和笔记内容
- **THEN** 系统必须提炼并输出内容定位卡，包含：
  - 账号定位
  - 目标人群
  - 笔记类型（种草/测评/经验分享/教程/避坑/日常/人设经营/商业转化）
  - 发布目的（曝光/涨粉/收藏/咨询/转化）
  - 核心卖点 / 核心观点
  - 用户读完后应产生的动作
  - 当前内容与账号定位匹配度（高/中/低）
- **AND** 如果信息缺失，使用"未提供"或"基于文本推测"

### Requirement: 风险分类精细化

The system SHALL distinguish between compliance risk and traffic risk:

#### Scenario: 违规风险判定
- **WHEN** 检测到可能触发平台审核的内容
- **THEN** 明确标注为"违规风险"，包括：
  - 敏感词、违禁词、极限用语
  - 强导流、诱导交易
  - 医疗/金融/教育违规承诺
  - 虚假宣传、夸大功效

#### Scenario: 流量风险判定
- **WHEN** 检测到可能影响自然推荐的内容
- **THEN** 明确标注为"流量风险"，包括：
  - 内容和账号定位偏离
  - 标题过于广告化
  - 开头没有痛点
  - 用户画像不清晰
  - 标签过泛或不相关
  - 过度堆叠关键词
  - 缺少真实体验感
  - 商业目的过强
  - 内容没有收藏价值
  - 话题与目标人群兴趣不匹配

### Requirement: 封面文案检测

The system SHALL evaluate cover text effectiveness:

#### Scenario: 封面文案检测
- **WHEN** 用户提供封面文案或图片文字
- **THEN** 检测以下维度：
  - 是否一眼看懂
  - 是否过度承诺
  - 是否信息过多（建议8-16字）
  - 是否符合笔记内容
  - 是否能吸引目标人群
  - 是否有违规词或极限词
  - 是否适合小红书阅读场景
- **AND** 提供3个优化版本建议

### Requirement: 评论区与互动引导检测

The system SHALL detect comment section risks:

#### Scenario: 评论区导流检测
- **WHEN** 检测笔记内容
- **THEN** 识别以下高风险表达：
  - "评论1发你"、"私信领"、"加我拿"
  - "主页有联系方式"、"需要的扣1"
  - "免费送资料包"
  - 诱导点赞收藏、诱导交易
- **AND** 提供合规的替代表达

### Requirement: 特殊行业处理规则

The system SHALL provide industry-specific detection rules:

#### Scenario: AI工具/SaaS/API平台类
- **WHEN** 检测到AI工具、SaaS、API平台相关内容
- **THEN** 重点检查：
  - 是否过度承诺效果
  - 是否存在"全网最低""官方指定""无限调用"等风险表达
  - 是否过度技术化，导致理解门槛高
  - 是否缺少具体使用场景
- **AND** 建议多用"适合什么场景""我怎么用""省去了哪些麻烦"

#### Scenario: 医美/减肥/健康类
- **WHEN** 检测到医美、减肥、健康相关内容
- **THEN** 重点检查医疗承诺、疗效表达、体重结果承诺、药品推荐
- **AND** 必须提醒"个人经验不能替代医疗建议"

#### Scenario: 教育/副业/赚钱类
- **WHEN** 检测到教育、副业、赚钱相关内容
- **THEN** 重点检查收益承诺、速成话术、保证结果、诱导报名
- **AND** 建议用"适合参考""个人经验""降低试错成本"

#### Scenario: 品牌种草/商业合作类
- **WHEN** 检测到品牌种草、商业合作相关内容
- **THEN** 重点检查是否过度广告化、缺少真实体验、过度堆卖点
- **AND** 建议先写用户痛点，再写使用体验，最后自然带出产品

### Requirement: 标准化输出模板

The system SHALL use a standardized 9-step output template:

#### Scenario: 输出结构
- **WHEN** 输出检测结果
- **THEN** 必须按以下顺序输出：
  1. 总体结论（风险等级、是否建议发布、一句话判断）
  2. 内容定位卡
  3. 核心问题摘要
  4. 敏感词与高风险表达检测
  5. 账号定位与人群匹配度
  6. 自然流影响因素判断
  7. 修改建议（标题/正文/封面/标签）
  8. 优化后版本
  9. 发布前检查清单

### Requirement: 信息不足处理流程

The system SHALL handle insufficient information gracefully:

#### Scenario: 信息不足时
- **WHEN** 用户只提供笔记文案，缺少账号定位
- **THEN** 最多询问3-5个关键问题：
  1. 这个账号主要做什么方向？
  2. 目标人群是谁？
  3. 这篇笔记想实现什么目的？
  4. 这篇笔记属于什么类型？
  5. 账号之前主要发哪些内容？
- **AND** 如果用户不想补充，继续完成基础检测，并标注"账号匹配度判断信息不足"

### Requirement: 修改原则

The system SHALL follow 5 modification principles:

#### Scenario: 修改文案
- **WHEN** 提供修改建议
- **THEN** 遵循以下原则：
  1. **降低风险**：绝对表达→相对表达，承诺表达→体验表达，营销表达→场景表达
  2. **增强匹配**：让标题、开头、正文、标签都服务同一个目标人群
  3. **提升自然感**：减少官方口吻，增加真实使用语境
  4. **保留核心卖点**：修改时不要过于保守，保留吸引力和转化价值
  5. **给出可直接替换的版本**：每个关键问题都应给出可复制使用的替代表达

## MODIFIED Requirements

### Requirement: 输出格式规范

The system SHALL enhance output format with risk classification:

#### Scenario: 风险等级表达
- **WHEN** 输出风险等级
- **THEN** 使用以下表达：
  - 低风险：内容整体安全，只有轻微表达优化空间
  - 中风险：存在部分敏感表达、营销感、标题夸张、定位偏移
  - 高风险：存在明显违规表达、强导流、夸大承诺
  - 建议规避 / 建议替换
  - 需要结合账号历史数据判断
- **AND** 避免使用"必限流""一定违规""一定会被封"等绝对化表达

## REMOVED Requirements

无移除需求，本次为增强型优化。
